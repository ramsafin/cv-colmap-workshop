# 01. Подготовка окружения: Docker, импорт образа, запуск контейнера

Здесь описано, как на **своём компьютере** подготовить одно и то же окружение, в котором дальше идёт весь воркшоп по COLMAP.

## Как пользоваться этим файлом

1. Установите программы из раздела **«Что установить на своём компьютере»** ниже.
2. Положите проект на диск и при необходимости положите видео `sample.mp4` (см. раздел **«Структура проекта»**).
3. Выберите **один** путь: **вариант A** (импорт готового образа) **или** **вариант B** (сборка образа).
4. Запустите контейнер (раздел **«Запуск контейнера»**) и выполните **быструю проверку** (`smoke_test.sh`).
5. Перейдите к **следующему шагу** в конце файла — извлечению кадров из видео.

**Хост** — это ваш компьютер с Windows, macOS или Linux. 

**Контейнер** — изолированная среда с уже установленными программами; внутри неё вы работаете в каталоге `/workspace`, который является **копией папки проекта** с диска (папка с файлом `compose.yaml` подставляется в контейнер автоматически).

## Цель этапа

На этом шаге нужно:

1. получить **Docker-образ** воркшопа (готовый файл-архив образа или сборка из исходников);
2. **импортировать** готовый образ **или** **собрать** его вручную;
3. **запустить контейнер** (одноразовая сессия с оболочкой `bash`);
4. проверить, что внутри контейнера доступны программы:
   - `colmap`
   - `ffmpeg`
   - `python3`
5. убедиться, что папки с данными и входное видео **видны** из контейнера и в них **можно писать**.

---

## Что установить на своём компьютере

На вашем компьютере должны быть установлены:

- **Docker** — среда для запуска контейнеров;
- **Docker Compose** — утилита для сборки и запуска по файлу [`compose.yaml`](../compose.yaml) (на Windows и macOS обычно входит в **Docker Desktop**; на Linux может понадобиться отдельная установка плагина Compose).

### Поддерживаемые платформы

- Linux (Ubuntu)
- macOS (Apple Silicon / ARM)
- Windows: **Docker Desktop** + **WSL2** + дистрибутив **Ubuntu** внутри WSL (команды из этого документа выполняйте в терминале Ubuntu, а не в `cmd.exe`).

---

## Структура проекта

Предполагается, что каталог с материалами воркшопа уже есть на диске (клон репозитория или распакованный архив) и выглядит так (пути — **относительно корня проекта**, там же где лежит `compose.yaml`):

```text
cv-colmap-workshop/
├─ README.md
├─ compose.yaml
├─ docker/
├─ docs/
├─ scripts/
└─ data/
   ├─ video/
   │  └─ sample.mp4 # скачайте или скопируйте с флешки преподавателя
   ├─ images/
   └─ workspace/
```

Перед первым запуском контейнера убедитесь, что на **хосте** (на диске вашего компьютера, не внутри Docker) существует файл:

```text
data/video/sample.mp4
```

Если видео ещё нет, его можно взять у преподавателя или скачать примеры с сайта [Tanks and Temples](https://www.tanksandtemples.org/download/) (англ. название набора данных; для воркшопа подойдёт ролик в формате, который открывает FFmpeg, обычно MP4).

---

## Вариант A. Импорт готового Docker-образа

**Рекомендуемый вариант** на занятии, если преподаватель раздаёт архив образа — быстрее, чем сборка из интернета.

### Шаг 1. Скопировать архив с образом

С флешки или из общей папки скопируйте **архив образа** (один файл), например:

```text
colmap-workshop-image.tar
```

в любую папку на диске (запомните путь к файлу `.tar`).

### Шаг 2. Импортировать образ в Docker

Откройте терминал, перейдите в папку, где лежит архив (или укажите полный путь к файлу вместо имени ниже), и выполните:

```bash
docker image load -i colmap-workshop-image.tar
```

Команда **загружает** сохранённый образ в локальное хранилище Docker.

### Шаг 3. Проверить, что образ появился

```bash
docker images
```

В таблице должна появиться строка с образом воркшопа (имя **репозитория** образа и **тег** — как в колонках ниже).

Пример ожидаемого фрагмента вывода (цифры и даты у вас будут другие):

```text
REPOSITORY          TAG       IMAGE ID       CREATED         SIZE
…                   …         …              …               …
colmap-workshop     latest    <hash>         <date>          <size>
```

---

## Вариант B. Сборка образа вручную

Используйте этот путь, если **нет** готового архива `.tar` или нужно пересобрать образ после правок в [`docker/Dockerfile`](../docker/Dockerfile).

Откройте терминал и перейдите в **корень проекта** — каталог, где лежит файл `compose.yaml` (команда `ls` должна показывать `compose.yaml`, папки `docker/`, `scripts/` и т.д.). Затем выполните:

```bash
docker compose build
```

Сборка может занять некоторое время (обычно требуется подключение к интернету).

**Внимание:** не рекомендуется использовать дозированный или мобильный интернет: при сборке скачиваются пакеты и зависимости, суммарный объём может измеряться **гигабайтами**.

После успешной сборки выполните `docker images` и убедитесь, что в списке есть образ с именем **`colmap-workshop`** и тегом **`latest`** (так задано в [`compose.yaml`](../compose.yaml) в корне проекта):

```bash
docker images
```

---

## Запуск контейнера

Снова находясь в **корне проекта** (рядом с `compose.yaml`), выполните:

```bash
docker compose run --rm workshop bash
```

Кратко, что означают части команды:

| Часть | Смысл |
|--------|--------|
| `docker compose run` | запустить одноразовый контейнер по правилам из `compose.yaml` |
| `--rm` | после выхода из контейнера **удалить** его (не копить лишние остановленные контейнеры) |
| `workshop` | имя **сервиса** в `compose.yaml` (роль «учебная машина воркшопа») |
| `bash` | открыть интерактивную **командную оболочку** внутри контейнера |

После запуска вы окажетесь **внутри контейнера**. 

Рабочий каталог по умолчанию — **`/workspace`**: это та же папка проекта с вашего диска, только путь внутри Linux-окружения контейнера.

Пример строки приглашения в терминале (имя и идентификатор контейнера будут свои):

```text
root@<идентификатор-контейнера>:/workspace#
```

Чтобы **выйти** из контейнера и вернуться на хост, введите `exit` или нажмите Ctrl+D.

---

## Что вы должны увидеть внутри контейнера

Compose настроен так, что **вся папка проекта с хоста** доступна в контейнере как `/workspace`. Проверьте, что существуют:

* `/workspace/README.md`
* `/workspace/docs/`
* `/workspace/scripts/`
* `/workspace/data/video/sample.mp4`
* `/workspace/data/images/`
* `/workspace/data/workspace/`

Проверка:

```bash
pwd
ls
ls /workspace
ls /workspace/data/video
```

Ожидается, что в списке файлов в `data/video` будет **`sample.mp4`**.

## Быстрая проверка окружения (smoke test)

**Дымовая проверка** (англ. *smoke test*) — короткий сценарий «всё ли на месте»: программы, пути, права на запись. Его удобно гонять после каждой переустановки Docker или смены папки проекта.

Все дальнейшие шаги воркшопа предполагают, что вы находитесь **в каталоге `/workspace`** внутри контейнера (так и будет сразу после `docker compose run … bash`) и что сценарии запускаются явно через оболочку:

```bash
bash scripts/имя_скрипта.sh
```

Запустите проверку:

```bash
bash scripts/smoke_test.sh
```

Если в блоке проверки COLMAP появилось предупреждение **`[WARN]`** про версию **не 3.9**, материалы воркшопа всё равно могут подходить, но при сбоях сначала сверьте опции с выводом `colmap help` для вашей сборки.

В [`docker/Dockerfile`](../docker/Dockerfile) при сборке образа через менеджер пакетов **pip** ставятся библиотеки Python (`numpy`, `pillow`, `scipy`, `opencv-python`, `scikit-image`) — ими можно пользоваться в **дополнительных** заданиях на Python внутри контейнера.

---

## Ручная проверка без скрипта (по желанию)

Если нужно проверить окружение вручную, без `smoke_test.sh`, выполните команды ниже **внутри контейнера** в `/workspace`.

### Проверка COLMAP

```bash
colmap help
```

Ожидаемый результат:

* команда выполняется без ошибки;
* отображается список доступных команд COLMAP.

### Проверка FFmpeg

```bash
ffmpeg -version
```

Ожидаемый результат:

* отображается версия FFmpeg.

### Проверка Python

```bash
python3 --version
```

Ожидаемый результат:

* выводится номер версии Python.

При желании проверьте также команду `python` (она может отсутствовать — это нормально). Для воркшопа достаточно **`python3`**, как в скрипте `smoke_test.sh`.

### Проверка видео

```bash
ls /workspace/data/video/sample.mp4
```

Ожидаемый результат:

* путь к файлу отображается без ошибки.

---

## Проверка прав записи

Внутри контейнера выполните:

```bash
touch /workspace/data/workspace/.write_test
ls /workspace/data/workspace
```

Затем удалите тестовый файл:

```bash
rm /workspace/data/workspace/.write_test
```

Если команды проходят без ошибки, в каталог `data/workspace` на диске (через смонтированный `/workspace`) **можно записывать** файлы — дальнейшие шаги COLMAP смогут сохранять базу и модели.

---

## Минимальный чек-лист

Считаем этап **подготовки окружения** успешным, если:

* Docker-образ импортирован или собран;
* контейнер запускается;
* `colmap help` работает;
* `ffmpeg -version` работает;
* `python3 --version` работает (так же проверяет `smoke_test.sh`);
* файл `sample.mp4` доступен внутри контейнера;
* каталоги `data/images` и `data/workspace` доступны для записи.

## Заметки по платформам

Команды `docker compose …` одинаковы; отличается только то, **где** вы открываете терминал (хост).

### Linux (Ubuntu)

Обычно достаточно:

```bash
docker compose build
docker compose run --rm workshop bash
```

или импорта готового образа и такого же запуска контейнера.

### macOS (чип Apple M-серии, ARM64)

Установите **Docker Desktop** для Mac. Затем в терминале перейдите в корень проекта на диске Mac и выполните:

```bash
docker compose run --rm workshop bash
```

### Windows (Docker Desktop и подсистема WSL2)

Кратко: Docker работает в связке с **Linux-подсистемой** (WSL2); команды из этого документа нужно вводить в **терминале Ubuntu внутри WSL**, а не в классической командной строке Windows.

Рекомендуемый порядок:

1. установить **Docker Desktop** и включить интеграцию с **WSL2** по инструкции Docker;
2. установить дистрибутив **Ubuntu** из Microsoft Store (или другой способ установки WSL);
3. открыть приложение **Ubuntu**, клонировать/скопировать проект в файловую систему WSL (например `~/projects/...`) — так меньше проблем с правами, чем при работе из `C:\...`;
4. из этого терминала перейти в корень проекта и выполнить:

```bash
docker compose run --rm workshop bash
```

## Типичные проблемы

### Проблема 1. Команда `docker compose` не найдена

Проверьте, что установлен **Docker Compose** (часть Docker Desktop или отдельный пакет на Linux).

Попробуйте:

```bash
docker compose version
```

### Проблема 2. Контейнер не запускается

Проверьте по очереди:

* **служба Docker** на компьютере запущена (Docker Desktop — в состоянии «Running», на Linux — `sudo systemctl status docker` или аналог);
* образ **`colmap-workshop:latest`** есть в выводе `docker images`;
* команда `docker compose run …` выполняется из **корня проекта** (рядом с `compose.yaml`).

### Проблема 3. В контейнере не видно `sample.mp4`

Проверьте:

* на **хосте** существует файл `data/video/sample.mp4` относительно корня проекта;
* папка проекта **смонтирована** в контейнер как `/workspace` (так задумано в `compose.yaml`; не меняйте пути, если не уверены);
* вы снова запускаете `docker compose` из **корня проекта**, а не из другой папки.

### Проблема 4. Нет прав на запись в `data/workspace`

Проверьте:

* права доступа к папке проекта на **хосте**;
* проект не открыт с носителя «только чтение»;
* не мешает ли антивирус или корпоративная политика **привязке каталога** (монтированию папки с диска в контейнер).

### Проблема 5. Команды `colmap` или `ffmpeg` не найдены

Возможные причины:

* образ собран с ошибкой или не до конца;
* загружен **не тот** архив образа;
* в контейнере сбита **переменная PATH** или запущен не тот образ (очень редко при штатном сценарии).

Обычно помогает пересборка (`docker compose build --no-cache`) или повторный импорт правильного `.tar`.
